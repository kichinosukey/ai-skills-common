# セッション引き継ぎプロンプト生成

会話の途中で context 切断 or 次セッション移行が必要なとき、
**読めばそのまま作業に入れる**レベルの引き継ぎ md を生成する。
会話要約ではなく、**次のAIへの作業指示書**として書く。

## トリガー

- 「引き継ぎ作って / handoff 生成 / 次セッション用にまとめて」
- 「continuation prompt」「bridge prompt」
- セッション終盤で作業未完了のまま context を渡したいとき

## 出力先

`/tmp/handoff_<topic-slug>_<yyyymmdd>.md`

- topic-slug: kebab-case、ASCII（例: `deck-redesign`, `api-migration`）
- yyyymmdd: 今日の日付（JST）
- 既存ファイルある場合は上書き確認

## 生成手順

### 1. 会話文脈から以下を抽出

- **作業ディレクトリ**: 絶対パス
- **主要ファイル**: 触っているファイルの一覧（役割コメント付き）
- **現状**: 作業全体の構成 / ステータス（どこまで進んだか）
- **要対応 (優先度順 A/B/C)**: 残タスクを優先度別に、判断根拠つきで書く
- **既確定事項**: 前任セッションで決着済みの論点（蒸し返し防止）
- **関連 memory / 参考資料**: MEMORY.md 配下や外部ドキュメントへの参照
- **最初にやること**: 次セッション着手時の最初の一手（1-3 ステップ）

### 2. スタイル規則

- 読み手は **次のAIセッション**（人間ではない）と想定
- 冒頭に「このファイルは前任セッションが次セッションに渡す作業指示。先頭から読み、指示を実行してください」旨を明記
- 各タスクに **なぜ** そうするかの根拠を1行添える（判断ブレ防止）
- 曖昧語（「うまく」「いい感じに」）禁止、具体的な数値・ファイル名・行数で書く
- 絵文字なし、敬語不要、技術用語そのまま
- commit hash / ファイル行数 / 未決箇所数など **検証可能な数値** を積極的に埋め込む

### 3. セクション構成（テンプレ）

```markdown
# 前任セッションからの引き継ぎ (<作業タイトル>)

このファイルは前任セッションが次セッションに渡す作業指示。先頭から読み、指示を実行してください。

## 作業ディレクトリ
<絶対パス>

主要ファイル:
- `xxx` (役割 / 行数 等)

## 現状の <N> <単位> 構成
<スライド構成 / モジュール構成 / ファイル構成 等>

## 要対応 (優先度順)

### A. <最優先カテゴリ>
- **<具体タスク>**: <根拠 / 判断基準>

### B. <次点>
### C. <関係者確認 等の blocking 待ち項目>

## 既確定事項 (前任セッションで決済済み)
- <蒸し返さない論点>

## 関連 memory (必要に応じ参照)
- `<memory file name>.md`

## 最初にやること
1. このファイルを読み終えたらユーザーに「引き継ぎ受領完了。どこから着手しますか」と確認
2. <推奨着手順>の提案
3. <現状確認コマンド / ファイル読み>
```

### 4. 書き終えたら

1. `/tmp/handoff_<topic>_<yyyymmdd>.md` に Write
2. ユーザーに次セッションで貼るべきプロンプト1行を提示:
   ```
   /tmp/handoff_<topic>_<yyyymmdd>.md を読んで指示に従って
   ```
3. 生成内容のサマリ（セクション見出しと行数）を短く報告

## 注意

- **会話要約ではない**。次のAIが「読んで実行」できる指示書として書く
- 未決事項は「要○○確認」等のマーカーで明示（曖昧に流さない）
- 機密情報は会話中の匿名化方針に従う
- 会話文脈が薄い場合は、AskUserQuestion で作業ディレクトリ / タイトル / 優先度判断を確認
- このスキル自体はコード変更を行わない（md 生成のみ）
